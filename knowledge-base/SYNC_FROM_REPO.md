# Skill: Sync from Repo

> **Scope:** This skill operates within the AI Operating System project only. It uses the connected GitHub repository as the source of truth for project state and synchronizes this project's knowledge base and instructions accordingly.
>
> **Type:** Workflow skill — Claude follows these instructions when triggered.
>
> **Runtime:** Claude.ai only (uses connected GitHub repo to read latest committed state).

---

## When to Use

Activate this workflow when the user says any of:
- "sync from repo," "sync state," "update KB from repo," "pull latest state"
- "what changed in the repo," "what's new in the codebase"
- "sync knowledge base," "update project state," "refresh KB"
- "is my KB up to date," "check for drift"
- At the start of a new working session if the user hasn't synced recently (suggest proactively: "I notice we haven't synced from the repo recently. Want me to run a quick sync?")
- After the user mentions they've been working in Claude Code

Do NOT activate mid-conversation when the user is focused on a different task. Offer to run it afterward.

---

## How It Works

This skill bridges two environments:
- **Claude Code** (this repo) — where code is written, deployed, and `/update-project-state` generates `PROJECT_STATE.md`
- **Claude.ai** (this project) — where planning, research, content, and communication happen

The repo's `knowledge-base/PROJECT_STATE.md` is the authoritative snapshot of what's built, what's pending, and what's blocked. It's generated by scanning the actual filesystem and git history — not by manual documentation. This skill reads that file and reconciles this project's knowledge base against it.

## Process

### Step 1: Fetch the Latest State from Repo

Browse the connected GitHub repository and read these files from the `knowledge-base/` directory:

**Required (always fetch):**
1. `knowledge-base/PROJECT_STATE.md` — The authoritative state snapshot
2. `knowledge-base/WORK_PROJECTS.md` — Operational project status
3. `knowledge-base/EVOLUTION_LOG.md` — Decision history and sprint log

**Conditional (fetch if the user reports changes in these areas, or if PROJECT_STATE.md references updates):**
4. `knowledge-base/TOOL_ECOSYSTEM_PLAN.md` — If tool/MCP changes
5. `knowledge-base/GCP_INFRA_CONFIG.md` — If infrastructure changes
6. `knowledge-base/INTERFACE_STRATEGY.md` — If interface/dashboard changes
7. `knowledge-base/DB_SCHEMA.md` — If database migrations were applied
8. `CLAUDE.md` (project root) — For current sprint focus and architecture context

Also check: `git log --oneline -10` equivalent — look at the most recent commit messages to understand what changed.

### Step 2: Parse the State Snapshot

From `PROJECT_STATE.md`, extract:

**A. Quick Reference Table**
- Status of each major area (Skills, MCP Gateway, Database, Cloud Functions, Dashboard, CI/CD)
- This becomes the PROJECT PULSE in morning briefs and session resumes

**B. Drift Report**
- **Overclaimed** — Things documented as done/built but that don't match reality
- **Undocumented** — Things found in the codebase but not in KB docs
- **Stale References** — Specific lines in KB docs or CLAUDE.md that need updating

**C. Blocked Items**
- What's waiting on external action, who needs to do it, what it impacts

**D. Next Actions**
- Priority-ordered list of what should happen next

**E. Change Log**
- What changed since the last state version

### Step 3: Compare Against Current KB

For each KB document in this project, determine if it's current or stale:

| KB Document | Repo Source | How to Check Staleness |
|-------------|-----------|----------------------|
| WORK_PROJECTS.md | `knowledge-base/WORK_PROJECTS.md` | Compare "Last updated" timestamp and "Status" line |
| EVOLUTION_LOG.md | `knowledge-base/EVOLUTION_LOG.md` | Compare latest entry number and date |
| TOOL_ECOSYSTEM_PLAN.md | `knowledge-base/TOOL_ECOSYSTEM_PLAN.md` | Compare "Last updated" line and module status table |
| GCP_INFRA_CONFIG.md | `knowledge-base/GCP_INFRA_CONFIG.md` | Compare secret count, service list |
| INTERFACE_STRATEGY.md | `knowledge-base/INTERFACE_STRATEGY.md` | Compare "Last updated" line |
| DB_SCHEMA.md | `knowledge-base/DB_SCHEMA.md` | Compare table count and migration list |
| PROJECT_STATE.md | `knowledge-base/PROJECT_STATE.md` | This is NEW — may not exist in KB yet |

**Classification for each doc:**
- **CURRENT** — KB version matches repo version (same timestamps, same content)
- **STALE** — KB version is older than repo version (needs re-upload)
- **MISSING** — Exists in repo but not uploaded to this project's KB
- **KB-ONLY** — Exists in KB but not in repo (legacy or manually created)

### Step 4: Generate the Sync Report

Present a structured sync report to the user:

---

**SYNC REPORT — [Date]**
**Repo state version:** [from PROJECT_STATE.md header]
**Last repo commit:** [hash + message]

**SYSTEM STATUS (from repo scan)**

| Area | Status | Change Since Last Sync |
|------|--------|----------------------|
| [from Quick Reference table] | | [NEW / CHANGED / SAME] |

**KB DOCUMENTS TO UPDATE**

| Document | Status | Action |
|----------|--------|--------|
| [doc name] | STALE | Re-upload from repo (key changes: [summary]) |
| [doc name] | MISSING | Upload from repo (new file) |
| [doc name] | CURRENT | No action needed |

**BLOCKERS & NEXT ACTIONS**
[From PROJECT_STATE.md Blocked and Next Actions sections — present as-is]

**PROJECT INSTRUCTIONS DELTA**
[If the repo's CLAUDE.md or PROJECT_STATE.md reveals that the project instructions in this Claude.ai project are stale, list the specific changes needed. See Step 5.]

**DRIFT ALERTS**
[From PROJECT_STATE.md Drift Report — anything the user should know]

---

### Step 5: Generate Project Instructions Updates

Read the current project instructions for this Claude.ai project (the user can paste them, or you reference what you know from conversation context). Compare against what the repo's state implies:

Check these specific areas in the project instructions:
1. **Skill count** — Does it match the actual count in `.claude/skills/`?
2. **Sprint focus** — Does it match the "Current Sprint" section in CLAUDE.md?
3. **What's built** — Does it match the LIVE section of PROJECT_STATE.md?
4. **MCP Gateway status** — Does it reflect all deployed modules correctly?
5. **KB document catalogue** — Does it list all KB documents, including any new ones?
6. **Pending work** — Does it match PLANNED and BLOCKED sections?

For each stale section, generate the updated text that the user can copy-paste into their project instructions.

Format as:

```
SECTION: [section name in project instructions]
CURRENT TEXT: "[what it says now — quote or summarize]"
UPDATED TEXT: "[what it should say]"
REASON: [what changed]
```

### Step 6: Help Execute the Sync

After presenting the report, guide the user through execution:

1. **For STALE KB docs:** "Download [filename] from the repo's knowledge-base/ directory and re-upload it here, replacing the old version."
   - If only 1-2 docs need updating, offer to show the full updated content in an artifact so the user can copy-paste instead of downloading.

2. **For MISSING KB docs:** "Upload [filename] from the repo — it's a new document that didn't exist before."

3. **For project instructions:** "Here are the lines to update in your project instructions: [copy-pasteable text]"

4. **For new skills:** If new skill files exist in the repo's `.claude/skills/` that aren't in this project's KB, list them and offer to display their content for upload.

After the user confirms they've made changes, acknowledge and note the sync is complete.

---

## Output Format

The sync report should be scannable in 90 seconds. Use tables for document status. Use code blocks for copy-pasteable instruction updates. Keep explanations to one line per item.

If only 1-2 things changed, keep the report to a short paragraph + the specific changes. Don't produce a full report when nothing is stale.

If everything is current, say: "KB is in sync with the repo (state version vN, committed [date]). No updates needed." and stop.

---

## Quality Rules

- **The repo is always the source of truth.** If the KB doc and repo disagree, the repo version wins. The repo state is generated by scanning the actual filesystem, not by manual documentation.
- **Never guess at repo contents.** If you can't access a file from the connected repo, say so and ask the user to paste the file contents instead.
- **Be specific about what changed.** "WORK_PROJECTS.md is stale" is useless. "WORK_PROJECTS.md: Status line says 'stubs deployed' but repo says 'fully implemented, blocked on OAuth'" is useful.
- **Don't overwhelm.** If many docs changed, prioritize: PROJECT_STATE.md first (most important — it's the authoritative snapshot), then WORK_PROJECTS.md, then EVOLUTION_LOG.md, then everything else.
- **Project instructions are high-leverage.** A stale project instruction affects every future conversation. Prioritize instruction updates over KB doc updates.
- **Skills are KB docs.** If a new skill was created in `.claude/skills/`, it needs to be uploaded to this project's KB as a document. List the file path so the user can download it from the repo.
- **Track sync state.** At the end of a successful sync, note the state version (from PROJECT_STATE.md) and date. Reference this in future syncs to determine what changed.
- **Suggest proactively.** If the user mentions they just finished a Claude Code session, ask: "Want me to sync from the repo to pick up any changes?"

## Relationship to /update-project-state

These two skills form a sync loop:

```
Claude Code                          Claude.ai
    │                                    │
    ├─ /update-project-state             │
    │   ├─ Scans filesystem              │
    │   ├─ Checks git history            │
    │   ├─ Detects drift                 │
    │   └─ Writes PROJECT_STATE.md       │
    │        │                           │
    │        └──── git commit + push ────┤
    │                                    │
    │                         /sync-from-repo
    │                           ├─ Reads PROJECT_STATE.md from repo
    │                           ├─ Compares against KB docs
    │                           ├─ Generates sync report
    │                           ├─ Updates KB docs
    │                           └─ Updates project instructions
    │                                    │
    └──── next Claude Code session ──────┘
```

**The user's workflow:**
1. Work in Claude Code → build things → run `/update-project-state` → commit + push
2. Switch to Claude.ai → say "sync from repo" → review sync report → apply updates
3. Continue working in Claude.ai with fresh context
4. Repeat

This ensures both interfaces always operate with accurate, verified project state.
