# Review Project (Phase 4)

The final quality gate. This skill runs AFTER the build is complete and BEFORE delivery. It catches both code quality issues AND design slop that the build agents missed.

Two reviewers run in sequence:
1. **Code Reviewer** — Build, types, lint, security, patterns
2. **Design Reviewer** — Aesthetic direction compliance, anti-slop, differentiation check

Both must pass before the project is declared ready to ship.

## Before Starting

1. Read `${CLAUDE_PLUGIN_ROOT}/context/engineering-preferences.md`
2. Locate the project codebase (from manifest.json `code_path` or current working directory)
3. Load the spec files from `forge-projects/{slug}/specs/`:
   - `design-direction.md` — for the design review
   - `prd.md` — for requirements coverage
   - `architecture.md` — for stack alignment
4. Read the `build-report.md` if it exists (generated by scaffold-app)

## Part 1: Code Review

### 1a. Build Verification
```bash
npm run build 2>&1
# Capture exit code — build failure is ALWAYS blocking
```

If build fails:
- Report exact errors with file:line references
- Suggest specific fixes
- Mark as **BLOCKING — cannot ship**

### 1b. Type Safety
```bash
npx tsc --noEmit 2>&1
```

Scan for:
- `any` types (each one is a violation)
- `@ts-ignore` or `@ts-nocheck` comments
- Missing type definitions for function parameters or returns
- Untyped API responses

For new forge projects: type errors are **blocking**.
For refine operations on existing projects: type errors are **warnings**.

### 1c. Lint Compliance
```bash
npm run lint 2>&1
```

Group violations by severity. Report counts per category.

### 1d. Pattern Analysis
Scan the codebase for these anti-patterns:

**Security:**
- Hardcoded secrets (grep for `sk-`, `ghp_`, `AIza`, `eyJ`, `password =`, `secret =`, API key patterns)
- CORS set to `*` in production configuration
- SQL string concatenation (should use parameterized queries)
- Missing auth middleware on routes that handle user data
- Sensitive data in API responses (passwords, tokens, internal IDs)

**Code Quality:**
- `console.log` statements in production code (not test/dev files)
- Empty catch blocks (`catch (e) {}` or `catch {}`)
- TODO/FIXME comments that should have been resolved
- Unused imports or variables
- Functions longer than 50 lines without decomposition
- Deeply nested conditionals (3+ levels)

**UX Completeness:**
- Missing error boundaries on page components (Next.js `error.tsx`)
- Missing loading states for async operations (Next.js `loading.tsx`)
- Missing 404/not-found handling
- Forms without validation feedback
- Buttons without disabled states during loading
- Missing mobile viewport meta tag
- Non-responsive layouts (fixed pixel widths without responsive alternatives)

**Infrastructure:**
- Missing `.env.example` entries for environment variables used in code
- Dockerfile issues (running as root, missing multi-stage, oversized image)
- Missing health check endpoint (`/health` or `/api/health`)

### 1e. Documentation Check
Verify existence and completeness:

| Document | Required Sections | Status |
|----------|------------------|--------|
| **README.md** | Purpose, Quick Start (3 steps), Architecture overview, Environment variables, Deployment guide | |
| **.env.example** | Every env var used in code, with descriptions | |
| **CLAUDE.md** | Project overview, tech stack, coding standards, directory structure | |

### 1f. Requirements Coverage
Cross-reference the PRD (if available):
- For each P0 requirement: does corresponding code exist?
- For each page in the page inventory: does the route exist with actual content?
- For each API endpoint in the spec: does the route exist with proper validation?

Report coverage as: `{covered}/{total} P0 requirements implemented`

## Part 2: Design Review

This part requires loading `forge-projects/{slug}/specs/design-direction.md`.

### 2a. Direction Compliance
Load the committed aesthetic direction statement from specs. Then verify:

- **Does the implemented UI match the stated archetype?**
  - If direction said "Operational Command" but the output is a generic card layout → FAIL
  - If direction said "Editorial Authority" but there's no typographic hierarchy → FAIL

- **Are the brand tokens actually applied?**
  - Check Tailwind config or CSS variables against the token system in design-direction.md
  - Every color in the token system should appear in the config
  - Fonts from the token system should be loaded and applied

- **Does the differentiation decision show up in the output?**
  - Load the specific mechanism from Step 5e of the spec
  - Search the codebase for evidence of its implementation
  - If the differentiation decision is "metric cards use a hairline left border" — does that exist?

### 2b. Anti-Slop Audit
Run the full anti-slop checklist against the implemented UI:

```
GENERIC AI PATTERNS FOUND:
□ Hero with gradient text + subtitle + two CTA buttons → [YES/NO] [file:line if found]
□ Three-column feature cards with icon + title + description → [YES/NO]
□ Stats row with big numbers and small labels → [YES/NO]
□ Cookie-cutter glassmorphism cards → [YES/NO]
□ Purple/blue gradient as primary visual → [YES/NO]
□ shadcn default theme without customization → [YES/NO]
□ Generic sidebar + main content without specific need → [YES/NO]
□ Inter/system-ui as only typeface (when spec says otherwise) → [YES/NO]
□ Identical card structures repeated without variation → [YES/NO]
□ Decorative animations without functional purpose → [YES/NO]
```

Each "YES" is a design violation that needs fixing.

### 2c. Visual Consistency Check
- Are colors consistent across all pages? (no rogue hex values outside the token system)
- Is typography hierarchy consistent? (same heading styles everywhere)
- Is spacing consistent? (same padding/margin patterns)
- Are interactive states defined? (hover, focus, active, disabled)
- Are empty/loading/error states designed (not just default browser states)?

### 2d. Accessibility Quick Scan
- Color contrast ratio: text on backgrounds meets WCAG AA (4.5:1 for body, 3:1 for large text)
- All interactive elements have focus-visible states
- Images have alt text
- Form inputs have labels
- Semantic HTML used (nav, main, article, section — not div soup)
- Skip-to-content link exists (for keyboard users)

## Decision Logic

**SHIP** if:
- Build passes
- Zero blocking security issues
- Design direction is aligned (minor drifts are warnings, not blockers)
- Anti-slop score >= 8/10 clean checks
- P0 requirements coverage >= 90%

**FIX REQUIRED** if:
- Build fails
- Any security issue found
- Design direction is violated (not just drifted)
- Anti-slop score < 8/10
- P0 requirements coverage < 80%

## After Review

If verdict is SHIP:
```
Project passes all quality gates. Ready for Phase 5 (Ship/Deploy).
```

If verdict is FIX REQUIRED:
```
{count} blocking issues found. Fix these before shipping:
{numbered list of blocking issues with fix suggestions}

Run review-project again after fixes.
```

## Quality Rules

- Build failure is ALWAYS blocking. No exceptions.
- Security issues are ALWAYS blocking. No exceptions.
- Design violations (anti-slop failures) ARE blocking for new forge projects. The whole point is avoiding slop.
- Be specific in reports. "Found type issues" is useless. "src/components/MetricCard.tsx:42 — props typed as `any`" is useful.
- The design review is not optional. If design-direction.md doesn't exist, flag it as a process violation and recommend running specify-project first.
- Don't be sycophantic. If the code is mediocre, say so. The report exists to catch problems, not to validate.
